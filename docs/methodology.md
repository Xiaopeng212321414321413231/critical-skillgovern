# Critical SkillGovern — 批判式技能治理方法论

> **核心洞察**：技能越多，路由复杂度指数级上升。系统不是能力不够，而是技能描述没有给模型足够的决策信号。
>
> 本方法论已在 **120+ 技能、40 个类别** 的实战中验证，覆盖 Hermes Agent、Claude Code、Codex 三大 AI 编程助手生态。

## 六阶段审查流水线

```
阶段 1: 路由诊断（Routing Diagnosis）— 发现问题
阶段 2: 批判性质询（Critical Questioning）— 深度分析
阶段 3: 用户审批（User Approval）— 决策拍板
阶段 4: 执行修复（Execution & Fix）— 精确施治
阶段 5: 二次验证（Verification）— 确保质量
阶段 6: 最终报告（Final Report）— 交付成果
```

## 阶段 1：路由诊断

### 检查清单

对每个技能进行 6 项路由健康检查：

| # | 检查项 | 症状 | 严重度 |
|---|--------|------|--------|
| 1 | 模板残余 | 描述中有 .join、kw[: 等 Python 字面量 | 严重 |
| 2 | 中英混合 | 触发场景为英文（专有名词除外） | 严重 |
| 3 | 重复描述 | 触发场景与输出目标几乎相同 | 建议优化 |
| 4 | 弱负样本 | 负样本太泛，未指向替代技能 | 建议优化 |
| 5 | 单层 tag | tags 只有一层 | 可选优化 |
| 6 | 缺关联 | related_skills 为空或缺少跨类链接 | 可选优化 |

### 通用路由结构

对于任何平台的技能，都应具备 4 层描述信息：

```
【触发场景】什么情况下触发
【输入特征】用户输入中的关键词/意图
【输出目标】最终产出什么
【不适用场景】什么情况不要用
```

## 阶段 2：批判性质询

### 5W1H 质问框架

- **What**: 这个技能的功能是什么？如果不是这样会怎样？
- **Why**: 为什么分类在这里？为什么不放别的类别？
- **Who**: 谁会使用这个技能？谁会误触发？
- **When**: 什么时候触发正确？什么时候会误触发？
- **Where**: 这个分类位置合理吗？
- **How**: 怎么执行这个技能？怎么验证？

### 逆向思维

```
正常 → 异常
├─ 正常触发 → 边界触发
├─ 正常流程 → 异常流程
└─ 正常环境 → 异常环境

确定 → 假设
├─ 用户会正确表达 → 用户会模糊表达
├─ 关键词会匹配 → 关键词不会匹配
└─ 技能会正确执行 → 技能会错误执行
```

### "那又怎样" 追问法

```
发现问题 → 那又怎样？
├─ 影响一个技能 → 那又怎样？还影响什么？
├─ 影响一个类别 → 那又怎样？还影响谁？
└─ 影响整个系统 → 那又怎样？影响多大？
```

## 阶段 3：用户审批

将阶段 1+2 的发现整理为决策清单：

```
严重:
  [ ] 模板残余 → 修复
  [ ] 全英触发 → 中文化
建议:
  [ ] 弱负样本 → 指向替代技能
可选:
  [ ] 缺关联 → 补充上下游链接
```

**审批原则：**
- 严重 → 几乎总是通过
- 建议 → 需要用户判断
- 可选 → 尊重用户意愿

## 阶段 4：执行修复

### 核心原则：只改路由，不碰本体

| 可修改 | 不可修改 |
|--------|----------|
| 描述/触发条件 | 功能代码、命令 |
| 关键词/tags | API 参数 |
| related_skills | 配置模板 |
| 负样本 | 文件结构 |

## 阶段 5：二次验证

执行修复后再次检查：

- [ ] 模板残余是否已清除？
- [ ] 触发场景是否已中文化？
- [ ] 输入特征是否覆盖自然语言变体？
- [ ] 负样本是否已指向替代技能？
- [ ] tags 是否已优化？
- [ ] related_skills 是否有跨类链接？
- [ ] YAML 语法是否正确？

## 阶段 6：最终报告

```
=== 技能审查报告 ===
技能: example-skill
类别: media/bilibili
审查结果: 5/6 项通过

严重 模板残余 → 已修复
严重 全英触发 → 已修复
建议 弱负样本 → 已修复
建议 重复描述 → 用户选择不修改
可选 单层 tag → 已修复
可选 缺关联 → 已修复

二次验证通过
```

## 自动化触发场景

1. 安装新技能后 → 自动审查
2. 编写新技能 → 创建时套用路由标准
3. 批量导入后 → 批量审查 + 优化
4. 系统升级前 → 全量审计
5. 发现误触发后 → 定向修复

## 方法论优势

- 跨平台通用
- 120+ 技能实战验证
- 可量化
- 渐进式
- 低侵入
