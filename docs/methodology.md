# Critical SkillGovern — 批判式技能治理方法论

> **核心洞察**：技能越多，路由复杂度指数级上升。系统不是能力不够，而是技能描述没有给模型足够的决策信号。
>
> 本方法论已在 **120+ 技能、40 个类别** 的实战中验证，并持续从审查实践中迭代升级。

## 七阶段审查流水线

[V2 新增] 可执行性检查、平台兼容性扫描（只标注不修改）、标签展示审批、能力发散挖掘、代码保护审计、底线原则。

---

## 阶段 1：路由诊断

### 基础检查清单

| # | 检查项 | 严重度 |
|---|--------|--------|
| 1 | 模板残余 | 描述中有 `.join`、`kw[:` 等 Python 字面量 | 🔴 |
| 2 | 中英混合 | 触发场景为英文（专有名词除外） | 🔴 |
| 3 | 重复描述 | 触发场景与输出目标几乎相同 | 🟡 |
| 4 | 弱负样本 | 负样本太泛，未指向替代技能 | 🟡 |
| 5 | 单层 tag | tags 只有一层 | 🟢 |
| 6 | 缺关联 | related_skills 为空 | 🟢 |

### 可执行性检查（V2 新增）

检查所有引用命令对应的脚本文件是否存在：

- ✅ 脚本存在 → 引用正常
- ⚠️ 文件名与命令名不同 → 需标注
- ❌ 脚本不存在 → 死引用，标记为严重

### 平台兼容性扫描（V2 新增）

扫描跨平台陷阱，只标注不修改：

| 假设 | 案例 | 标注 |
|------|------|------|
| Linux 路径 | `/tmp/`、`/var/` | Windows 需 `$TEMP` |
| Unix 命令名 | `python3`（Windows 只有 `python`） | 需要 fallback |
| 环境变量 | `$HOME` | Windows `$USERPROFILE` |

**原则：发现问题后只标注不修改**，下载者自行适配。

---

## 阶段 2：分类打标

### 标签必须展示给用户审批（V2 新增）

**流程**：拟定 tags → 展示给用户查看 → 确认/调整 → 写入。

```
=== 标签审批 ===
技能: xxx
  tags:        [category, subcategory]
  related:     [skill-a, skill-b]
  ✅ 确认 / 🔄 改一下
```

### 用户验证分类结果（关键环节）

分类完成后列出所有 tags → 用户过目 → 确认或调整。

---

## 阶段 3：批判性质询 + 能力挖掘

### 3D：能力发散挖掘（V2 新增）

跳出作者描述，从四个维度拷问每个技能还能产出什么：

```text
素材维度 — 主题/字体/配色还能做海报？封面？Banner？
技术管线维度 — 渲染管线还能做批量图？帧渲染？API？
数据维度 — 处理的数据还能做可视化？自动配图？
组合维度 — 能和其他 skill 配合出什么？
```

每条发现必须附带改进建议。

### 3E：代码保护审计（V2 新增）

检查源文件完整性。原作者的代码一行不改，扩展以独立文件追加。

---

## 阶段 4：用户审批

每条决策项包含问题、现状、方案、建议。

```
🔴 严重：
  [ ] 命令缩写未展开（9处）
      → 展开为完整路径
🟡 能力挖掘：
  [ ] 发现：视觉主题引擎可用于海报
      → 加独立扩展脚本
🟢 兼容性标注：
  [ ] `/tmp/` Windows 不兼容
      → 加标注即可
```

---

## 阶段 5：执行修复

### 代码扩展执行规范（V2 新增）

1. 原文件绝对不动
2. 新文件 `scripts/_extensions/ext_*`
3. 文件头注明非原作者代码
4. 不修改任何依赖文件

---

## 阶段 6：二次验证

新增：工具兼容性标注已添加？原代码未被修改？扩展文件遵循命名规范？

---

## 阶段 7：最终报告 + 扩展建议档案（V2 新增）

输出审查报告 + 扩展建议档案。

---

## 底线原则（V2 新增）

1. **工具适应不是 skill 的责任** — 平台差异只标注不修改
2. **标签必须经过用户审批** — 拟定后强制展示
3. **原代码一行不动** — 扩展用独立文件
4. **每项发现必须跟改进方向**
5. **发散挖掘隐藏能力**

---

## 🆕 附录：平台无关的分类 Prompt 模板（强约束版）

> **为什么必须强约束**：实战证明（WorkBuddy 29-skill 分类失败案例），弱 prompt 会导致 LLM 自创类别、重复归类、垃圾技能混入。以下 prompt 经过加固，可直接复制到任何平台的 n8n 工作流中。

### 完整的可用 Prompt 文件

独立保存于 `docs/prompts/classify-skills-v2.txt`，可直接嵌入 n8n LLM 节点的 System Prompt。

### Prompt 核心加固点

| 加固项 | 解决的问题 |
|--------|-----------|
| **角色定义** | "你没有权力创建新类别" |
| **铁律 5 条** | ❌禁止新类 ❌禁止重复 ❌禁止拆分 ✅归入实用工具 ✅严格格式 |
| **过滤规则** | 跳过纯数字 ID、空 description、系统保留名 |
| **Few-Shot 示例** | 8 个正确示例 + 4 个错误示例（含反模式） |
| **歧义消解 3 层优先级** | 核心产出 → 目标用户 → 上游依赖 |
| **输出格式约束** | "不要加说明、不要加序号、不要加 markdown" |

### 使用方法（n8n 工作流）

1. 在 n8n 的 LLM 节点 → System Prompt 中粘贴 `docs/prompts/classify-skills-v2.txt` 的内容
2. 在 User Message 中传入技能列表（每行一个技能名）
3. 解析 LLM 输出（格式: `技能名: 类别 → 子类`）

### 反模式对照表

| 错误行为 | 典型案例 | 修复方式 |
|---------|---------|---------|
| 自创类别 | "内容创作/开发工具/知识管理" | prompt 加"你没有权力创建新类别" |
| 重复归类 | playwright-mcp 同时出现在 2 个类 | prompt 加"禁止重复归类" |
| 垃圾技能混入 | skill_2053082144792322048 | prompt 加过滤规则 |
| 格式混乱 | 带序号/说明/解释 | prompt 加"不要加说明、不要加序号" |

> ⚠️ 如果分类结果仍有问题，先检查 LLM 输出中是否有以上反模式，再加固对应的 prompt 约束
