# Critical SkillGovern — 批判式技能治理方法论

> **核心洞察**：技能越多，路由复杂度指数级上升。系统不是能力不够，而是技能描述没有给模型足够的决策信号。
>
> 本方法论已在 **120+ 技能、40 个类别** 的实战中验证，并持续从审查实践中迭代升级。

## 七阶段审查流水线

[V2 新增] 可执行性检查、平台兼容性扫描（只标注不修改）、标签展示审批、能力发散挖掘、代码保护审计、底线原则。

---

## 阶段 1：路由诊断

### 基础检查清单

| # | 检查项 | 严重度 |
|---|--------|--------|
| 1 | 模板残余 | 描述中有 `.join`、`kw[:` 等 Python 字面量 | 🔴 |
| 2 | 中英混合 | 触发场景为英文（专有名词除外） | 🔴 |
| 3 | 重复描述 | 触发场景与输出目标几乎相同 | 🟡 |
| 4 | 弱负样本 | 负样本太泛，未指向替代技能 | 🟡 |
| 5 | 单层 tag | tags 只有一层 | 🟢 |
| 6 | 缺关联 | related_skills 为空 | 🟢 |

### 可执行性检查（V2 新增）

检查所有引用命令对应的脚本文件是否存在：

- ✅ 脚本存在 → 引用正常
- ⚠️ 文件名与命令名不同 → 需标注
- ❌ 脚本不存在 → 死引用，标记为严重

### 平台兼容性扫描（V2 新增）

扫描跨平台陷阱，只标注不修改：

| 假设 | 案例 | 标注 |
|------|------|------|
| Linux 路径 | `/tmp/`、`/var/` | Windows 需 `$TEMP` |
| Unix 命令名 | `python3`（Windows 只有 `python`） | 需要 fallback |
| 环境变量 | `$HOME` | Windows `$USERPROFILE` |

**原则：发现问题后只标注不修改**，下载者自行适配。

---

## 阶段 2：分类打标

### 标签必须展示给用户审批（V2 新增）

**流程**：拟定 tags → 展示给用户查看 → 确认/调整 → 写入。

```
=== 标签审批 ===
技能: xxx
  tags:        [category, subcategory]
  related:     [skill-a, skill-b]
  ✅ 确认 / 🔄 改一下
```

### 用户验证分类结果（关键环节）

分类完成后列出所有 tags → 用户过目 → 确认或调整。

---

## 阶段 3：批判性质询 + 能力挖掘

### 3D：能力发散挖掘（V2 新增）

跳出作者描述，从四个维度拷问每个技能还能产出什么：

```text
素材维度 — 主题/字体/配色还能做海报？封面？Banner？
技术管线维度 — 渲染管线还能做批量图？帧渲染？API？
数据维度 — 处理的数据还能做可视化？自动配图？
组合维度 — 能和其他 skill 配合出什么？
```

每条发现必须附带改进建议。

### 3E：代码保护审计（V2 新增）

检查源文件完整性。原作者的代码一行不改，扩展以独立文件追加。

---

## 阶段 4：用户审批

每条决策项包含问题、现状、方案、建议。

```
🔴 严重：
  [ ] 命令缩写未展开（9处）
      → 展开为完整路径
🟡 能力挖掘：
  [ ] 发现：视觉主题引擎可用于海报
      → 加独立扩展脚本
🟢 兼容性标注：
  [ ] `/tmp/` Windows 不兼容
      → 加标注即可
```

---

## 阶段 5：执行修复

### 代码扩展执行规范（V2 新增）

1. 原文件绝对不动
2. 新文件 `scripts/_extensions/ext_*`
3. 文件头注明非原作者代码
4. 不修改任何依赖文件

---

## 阶段 6：二次验证

新增：工具兼容性标注已添加？原代码未被修改？扩展文件遵循命名规范？

---

## 阶段 7：最终报告 + 扩展建议档案（V2 新增）

输出审查报告 + 扩展建议档案。

---

## 底线原则（V2 新增）

1. **工具适应不是 skill 的责任** — 平台差异只标注不修改
2. **标签必须经过用户审批** — 拟定后强制展示
3. **原代码一行不动** — 扩展用独立文件
4. **每项发现必须跟改进方向**
5. **发散挖掘隐藏能力**

---

## 与旧流程的区别

| 旧流程 | V2 新增 |
|--------|---------|
| 仅路由检查 | + 可执行性检查 + 平台兼容性扫描 |
| 纯找问题 | + 能力发散挖掘 + 改进方向 |
| 可改源代码 | + 代码保护审计 + 底线原则 |

## 🆕 附录：平台无关的分类 Prompt 模板

> 以下模板可直接嵌入任何 AI Agent（WorkBuddy / Claude Code / Codex / ChatGPT）的 n8n 工作流或 system prompt 中，无需适配。

### 分类指令模板

```
任务：对以下技能集合进行分类。

分类标准（请严格使用以下 7 个大类，不要自创新类）：

🌐 信息采集 (information-collection)
├── social-media      — 社交媒体信息获取、内容采集
├── web-scraping      — 网页爬取、结构化提取
├── hot-trends        — 热点追踪、趋势监控
└── download          — 文件/媒体下载

✍️ 信息整理 (information-organization)
├── note-taking       — 笔记管理、知识库
├── mind-map          — 思维导图
└── slides            — 演示文稿、PPT

🔍 搜索与资源发现 (search-resource-discovery)
├── search            — 通用搜索、学术搜索
└── files             — 文件查找、资源发现

🤖 智能代理与自动化 (agent-automation)
├── agent             — AI 代理、IDE 代理
└── automation        — 工作流自动化

📊 开发与工具 (development-tools)
├── database          — 数据库
└── dev-tools         — 开发工具、调试器

🎥 多媒体 (multimedia)
├── video             — 视频处理
├── audio             — 音频处理
├── image             — 图片处理
└── design            — 设计、视觉

🔧 实用工具 (utilities)
├── 通用工具（不匹配以上任何类别时归入此类）

规则：
1. 每个技能只能分到 1 个类别下的 1 个子类（双向互斥）
2. 如果技能匹配多个类别，选最核心的功能
3. 如果技能完全不匹配以上任何类别，归入"实用工具"
4. 输出格式：

```
技能名: [大类] → [子类]
```

示例：
```
agent-reach: 信息采集 → social-media
bilibili-video-extractor: 信息采集 → social-media
dashi-ppt: 信息整理 → slides
obsidian: 信息整理 → note-taking
last30days: 实用工具 → 通用工具
```
```

将此模板保存为独立的 prompt 文件，WorkBuddy 的 n8n 工作流可在调用 LLM 分类时直接注入。

### 互斥规则

同一技能不能同时出现在多个类别中。当 LLM 犹豫时：

1. **看核心产出** — 产出是一张图？→ multimedia/image。产出是一份笔记？→ information-organization/note-taking
2. **看目标用户** — 给开发者用？→ development-tools。给所有人用？→ utilities
3. **看上游依赖** — 需要外部 API？→ information-collection。本地处理？→ multimedia
